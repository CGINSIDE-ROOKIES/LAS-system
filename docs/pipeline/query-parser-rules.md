# Query Parser 규칙 문서

> 대상 코드: `apps/backend/rag/rag_pipeline/query_parser/`
> 작성일: 2026-04-01

---

## 1. 역할

사용자 질문을 LLM(기본: `gemini-2.0-flash-lite`)으로 분석해 아래 4개 필드를 구조화한다.
파이프라인은 이 결과를 기반으로 검색 필터와 law 문서 보강 전략을 결정한다.

```python
@dataclass
class QueryParseResult:
    law_names: list[str]    # 정식 법령명 목록
    article_no: str         # 조문번호 (정규화됨)
    intent: str | None      # "normative" | "case_law" | "mixed" | None
    is_legal: bool          # 법률 무관 질문이면 False
    parser_fallback: bool   # 파싱 실패 시 True
```

---

## 2. `is_legal` — 법률 관련성 판별

| 값 | 의미 | 파이프라인 동작 |
|----|------|----------------|
| `true` | 법률 관련 질문 | 검색 + 생성 파이프라인 실행 |
| `false` | 법률 무관 질문 | 즉시 고정 안내 문구 반환 (검색 없음) |

반환되는 안내 문구:
> "저는 노동법·하도급법 전문 법률 Q&A 어시스턴트입니다. 법률 관련 질문을 해주시면 도움을 드릴 수 있습니다."

**`is_legal=false` 예시**
- "오늘 점심 뭐 먹을까요"
- 일반 상식, 날씨, 코딩, 요리 등 법률과 무관한 모든 질문

**파싱 실패 시 (`parser_fallback=true`) 기본값:** `is_legal=true` (안전하게 검색 진행)

---

## 3. `intent` — 질의 유형

답변에 필요한 근거 문서의 성격을 나타낸다.

### normative (규범형)
- **의미:** 법령 조문이 핵심 근거인 질문. 요건·기준·의무·절차 등을 묻는 경우.
- **파이프라인 영향:**
  - `law_names` 필터 적용
  - law 문서 최소 보장 강제 (`enforce=True`) → top_k 바깥에서도 law 문서 끌어옴
- **예시:**
  - "하도급 대금 지급 기한을 어기면 어떤 불이익이 있나요"
  - "연장근로 한도는 몇 시간인가요"
  - "육아휴직 신청 절차가 어떻게 되나요"

### case_law (판례형)
- **의미:** 판례·행정해석·결정례 등 실제 사례 기반 답변이 필요한 질문.
- **파이프라인 영향:**
  - `law_names` 필터 **해제** (None 처리) → 법령명 무관하게 전체 검색
  - law 문서 강제 보강 **해제** (`enforce=False`)
- **예시:**
  - "프리랜서로 계약한 인력이 근로자로 인정될 수 있나요"
  - "묵시적 근로계약 갱신 인정된 판례가 있나요"
  - "하청업체 소속 파견근로자의 사용자 책임 인정 사례"

### mixed (혼합형)
- **의미:** 법령 조문과 판례·해석 모두 필요한 질문.
- **파이프라인 영향:**
  - `law_names` 필터 적용 (normative와 동일)
  - law 문서 강제 보강 **해제** (`enforce=False`)
- **예시:**
  - "연장근로 수당 관련 분쟁 판례나 해석 사례가 있나요"
  - "최저임금 위반 시 처벌 기준과 실제 적용 사례"

### null (미분류 / 파싱 실패)
- **의미:** 파서가 intent를 확정하지 못한 경우 또는 `parser_fallback=true`.
- **파이프라인 영향:** normative와 동일하게 처리 (law 필터 + 강제 보강 적용)

**intent 별 파이프라인 동작 요약**

| intent | law_names 필터 | law 강제 보강 |
|--------|:-------------:|:------------:|
| `normative` | ✅ | ✅ |
| `mixed` | ✅ | ❌ |
| `case_law` | ❌ | ❌ |
| `null` | ✅ | ✅ |

## 4. `law_names` — 법령명 추출

### 인식 가능 법령 목록 (24개)

### 우선순위: 요청 값 vs 파서 결과

```
요청 바디에 law_names가 명시된 경우 → 파서 결과 무시, 요청 값 사용
요청 바디에 law_names가 null인 경우  → 파서 추출 결과 사용
```

### 목록 외 법령 처리

LLM이 목록에 없는 법령명을 출력하면 `_normalize_law_names()`에서 제거된다.  
즉 `law_names`는 항상 위 24개 중 일부이거나 빈 리스트(`[]`)다.

## 5. `article_no` — 조문번호 추출

- 질문에 조문번호가 명시된 경우 추출 (예: "제60조")
- 없으면 빈 문자열 `""`
- 정규화 규칙: 숫자로 시작하는 경우 `제` 접두어 자동 추가
  - `"17조"` → `"제17조"`
  - `"제17조"` → `"제17조"` (그대로)

> 현재 `article_no`는 검색 필터에 직접 사용되지 않고 로그 기록 용도로만 활용된다.

## 6. Few-shot 예시

LLM 프롬프트에 포함되는 4개의 예시:

| 질문 | law_names | article_no | intent | is_legal |
|------|-----------|------------|--------|----------|
| 하도급 대금 지급 기한을 어기면 어떤 불이익이 있나요 | `["하도급거래 공정화에 관한 법률"]` | `"제13조"` | `normative` | `true` |
| 프리랜서로 계약한 인력이 근로자로 인정될 수 있나요 | `[]` | `""` | `case_law` | `true` |
| 연장근로 수당 관련 분쟁 판례나 해석 사례가 있나요 | `["근로기준법"]` | `"제53조"` | `mixed` | `true` |
| 오늘 점심 뭐 먹을까요 | `[]` | `""` | `null` | `false` |

## 7. 파싱 실패 처리 (Fallback)

파싱이 실패하거나 LLM 응답에서 JSON을 추출할 수 없을 때:

- `strict_mode=false` (기본): 경고 로그 출력 후 fallback 결과 반환
- `strict_mode=true` (환경변수 `QUERY_PARSER_LLM_STRICT=true`): 예외 raise

**Fallback 결과**
```python
QueryParseResult(
    law_names=[],
    article_no="",
    intent=None,       # → normative와 동일하게 처리됨
    is_legal=True,     # 안전하게 검색 진행
    parser_fallback=True,
)
```

## 8. 환경변수

| 변수 | 기본값 | 설명 |
|------|--------|------|
| `QUERY_PARSER_LLM_PROVIDER` | `LLM_PROVIDER` → `gemini` | `gemini` 또는 `openai_compat` |
| `QUERY_PARSER_LLM_MODEL` | `LLM_MODEL` → provider 기본값 | 파서에 사용할 모델 |
| `QUERY_PARSER_LLM_URL` | `LLM_URL` 또는 provider 기본 URL | OpenAI-compatible `/chat/completions` endpoint 또는 Gemini endpoint override |
| `QUERY_PARSER_LLM_API_KEY` | `LLM_API_KEY` | 파서 LLM API 키 |
| `QUERY_PARSER_LLM_TIMEOUT` | `10` | 파서 API 타임아웃 (초). 짧게 설정해 latency 최소화 |
| `QUERY_PARSER_LLM_STRICT` | `false` | `true`로 설정 시 파싱 실패 예외 발생 |
