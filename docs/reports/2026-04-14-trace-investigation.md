## 1. 개요

웹앱에서 반환된 5개 근거 중 일부가 서로 매우 유사하게 보인 이유를 추적하고, 실제로 어떤 검색/융합 단계에서 중복성이 발생했는지 규명한다.

이번 조사에서는 코드 수정 없이 아래 두 축으로 원인을 복원하였다.

- Langfuse trace JSON 확인
- 운영 DB의 `qa_history` / `qa_sources` 및 현재 런타임 재현 결과 확인

> **주의:** 제공된 trace JSON에는 `query_parse`, `retrieval`, `fusion`, `ranking` 관측명이 있으나 실제 입력/출력 payload가 거의 비어 있어, trace 파일만으로는 최종 선택 문서를 직접 확인할 수 없었다. 동일 질문·시각대의 저장 이력을 DB에서 복원해 함께 분석했다.

## 2. 결론 요약

핵심 원인은 단일 버그가 아닌 아래 4가지가 겹친 결과다.

1. OpenSearch BM25가 같은 법조문 인용문을 공유하는 서로 다른 사건 청크 3개를 상위권으로 올렸다.
2. 이 3개는 서로 다른 `source_id`를 가진 별도 사건 청크라서 현재 dedup 로직에서 제거되지 않았다.
3. RRF 이후 ranking 단계에 텍스트 유사도 기반 다양화(diversity) 또는 near-duplicate 억제 로직이 없다.
4. 해당 사건 청크들이 `related_law_names`에 근로자퇴직급여 보장법을 포함하고 있어, 법령 필터를 걸어도 그대로 통과했다.

즉, 이번 현상은 "같은 문서가 중복으로 들어온 것"이 아니라, **서로 다른 헌법재판소 사건 청크들이 거의 같은 공무원연금법 제64조 / 시행령 제55조 구간을 담고 있어 결과적으로 near-duplicate처럼 보인 케이스**다.

또한 현재 런타임에서도 같은 질문으로 거의 동일한 Top-5가 재현되며, 유사 청크들은 모두 Qdrant가 아닌 **OpenSearch BM25 측 기여**로 유입됐다.

## 6. 왜 법령 필터를 걸어도 이런 사건이 통과했는가

이 질문은 파서가 근로자퇴직급여 보장법 필터를 잡지만, **공무원연금법 제목의 헌재 사건들이 그대로 통과**한다.

실제 OpenSearch에 저장된 해당 청크의 메타데이터:

```
root_law_name = 근로기준법
related_law_names = ["근로기준법", "근로기준법 시행령", "근로자퇴직급여 보장법", "근로자퇴직급여 보장법 시행령"]
```

문서의 실질 쟁점은 공무원연금법 사건이지만, 메타데이터 상으로 근로자퇴직급여 보장법과 관련된 사건으로도 태깅되어 있다. OpenSearch BM25는 `law_name`, `root_law_name`, `related_law_name`, `related_law_names` 중 **하나만 맞아도 통과**시키므로, 이 세 사건은 필터에서 탈락하지 않는다.

정리:

- 법령 필터는 "이 문서의 주된 법령"을 엄격히 거르지 않는다.
- "관련 있다고 한 번이라도 연결된 법령명"만 있어도 통과할 수 있다.
- 이번 케이스에서는 그 메타데이터가 너무 넓어, 질문 의도와 거리가 있는 공무원연금 사건들이 유입됐다.

## 7. 발생 메커니즘 상세

### 7.1 BM25가 넓게 잡는다

OpenSearch BM25 쿼리는 `search_text` 필드에 `operator: "or"`로 매칭한다. 질문어 일부만 강하게 걸려도 상위권에 노출될 수 있으며, "퇴직급여", "퇴직수당" 같이 반복 빈도가 높은 용어가 있으면 같은 구절을 공유하는 여러 문서가 함께 떠오르기 쉽다.

관련 코드: `retrieval/opensearch.py:50-87`, `retrieval/opensearch.py:90-145`

### 7.2 사건 청크의 search_text는 본문 인용문까지 포함한다

사건 본문을 청크로 쪼개고 그 본문을 그대로 `search_text`에 넣는 구조다. 여러 사건이 같은 조문을 길게 인용하면, BM25 관점에서는 각각이 독립적인 강매치 문서가 된다.

관련 코드: `legal_case_dataset_builder.py:68-99`, `legal_case_dataset_builder.py:394-431`

### 7.3 청크는 서로 다른 사건이면 모두 별도 문서다

사건 청크 ID 구조: `case_chunk::{canonical_case_id}::{chunk_index}`

이번 케이스의 3개는 모두 별도 사건이다:

- `case::detc::26493` — 2012헌바270
- `case::detc::29633` — 2013헌바205
- `case::detc::26294` — 2010헌마196

셋 모두 공무원연금법 제64조와 시행령 제55조 감액 규정을 유사하게 인용한 헌재 사건이다.

### 7.4 현재 dedup은 같은 source_id 계열만 제거한다

`normalize_source_id()`는 `__dup\d+$` suffix를 제거한 뒤 동일 source 여부를 판단한다. 다른 사건이지만 내용이 80% 유사한 경우는 제거되지 않는다.

관련 코드: `retrieval/common.py:187-218`

이 로직이 제거하는 경우:
- 같은 문서의 단순 중복
- `__dup0`, `__dup1` 같은 suffix만 다른 동일 source
- source_id가 없을 때의 동일 텍스트 해시 fallback

이번 3개는 source_id가 다르고 텍스트도 완전 동일하지 않아 모두 통과한다.

### 7.5 RRF와 ranking도 다양성을 보지 않는다

융합 후 ranking 단계는 점수 정렬과 law 문서 최소 개수 보강만 수행한다. 유사한 텍스트가 상위권에 몇 개 들어와 있는지, 같은 쟁점의 반복 문서인지, 새로운 정보가 있는지는 보지 않는다.

관련 코드: `generation/pipeline.py:230-315`, `retrieval/ranking.py:56-175`

## 8. 이번 케이스의 직접 원인

> 퇴직급여 질의에 대해 OpenSearch BM25가 동일한 감액 조문 인용문을 담은 여러 헌재 사건 청크를 상위로 올렸고, 이 청크들은 `related_law_names`에 근로자퇴직급여 보장법이 포함되어 필터도 통과했으며, 이후 dedup/ranking에 텍스트 유사도 기반 제거가 없어 최종 Top-5에 함께 남았다.

발생 순서:

1. 질문이 `normative`로 파싱됨
2. 근로자퇴직급여 보장법 필터가 잡힘
3. 공무원연금 사건 청크들이 `related_law_names` 때문에 필터를 통과함
4. BM25가 퇴직급여/퇴직수당 반복 문구를 강하게 잡아 유사 청크 3개를 상위권에 배치함
5. RRF + ranking이 다양화를 하지 않아 그대로 최종 5개에 남음

## 9. 추가 관찰

### 9.1 trace 파일만으로는 원인을 끝까지 볼 수 없다

이번 JSON export에는 observation 이름은 남아 있지만 핵심 payload가 비어 있었다. Langfuse export만으로는 "어떤 문서가 반환됐고 어떤 backend가 넣었는지"를 재구성할 수 없었다. 운영 DB의 `qa_sources`와 런타임 재현으로 보완했다.

### 9.2 이 문제는 기존 품질 이슈와도 연결된다

이번 현상은 주로 OpenSearch BM25와 dedup/diversity 문제였지만, 아래 기존 이슈와도 연결된다.

- 규범형 질문에 대한 law 우선 정책이 약함
- 법령 필터가 지나치게 넓은 관련성 메타데이터에 의존함
- retrieval 결과에 "새로운 정보성"보다 "점수"가 우선됨

이번 건은 단순한 예외 케이스가 아니라, **현재 RAG 설계가 유사 판례 군집을 쉽게 반복 노출하는 구조라는 신호**로 보는 편이 맞다.

## 10. 최종 판단

이번 trace의 이상 현상은 재현 가능했고, 원인도 비교적 명확했다.

- 중복처럼 보인 3개는 실제로는 서로 다른 사건 청크였다.
- 하지만 본문 인용 구간이 매우 유사해 사용자 관점에서는 사실상 중복 근거처럼 보였다.
- 이 유사 청크들은 OpenSearch BM25에서 유입되었고, 현재 dedup/ranking은 이를 걸러내지 못한다.
- 관련 법령 메타데이터가 넓게 붙어 있어 법령 필터도 충분히 좁혀주지 못했다.

따라서 이번 현상은 우발적인 UI 문제라기보다, **retrieval 설계상 near-duplicate 다양성 제어 부재와 broad metadata filtering이 만들어낸 품질 문제**로 판단된다.
